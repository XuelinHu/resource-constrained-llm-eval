Large language models have become practical tools for translation, question
answering, code assistance, and technical support. Their adoption in railway
engineering is promising because railway documents contain bilingual
terminology, maintenance regulations, and procedural knowledge that are
frequently queried in education and laboratory settings. However, practical
deployment is constrained by hardware. Many teams can access a single
workstation GPU but cannot rely on multi-GPU inference, large-scale
full-parameter fine-tuning, or remote proprietary services.

This paper studies that practical boundary directly. The target environment is
a single NVIDIA RTX 3090 GPU with 24GB of VRAM. Under this constraint, model
selection cannot be reduced to a public leaderboard score. A usable model must
fit reliably into memory, answer with acceptable latency, support low-bit
inference, and remain adaptable to local railway data. These constraints make
open models such as Qwen~\cite{qwen3,qwen25},
DeepSeek~\cite{deepseekllm,deepseekr1}, Phi~\cite{phi3}, Yi~\cite{yi2024open},
and Mistral~\cite{mistral7b} especially relevant, but they also require
consistent local measurement.

The work is positioned as an AI application and experimental evaluation paper,
not as a new model architecture. We build a railway bilingual dataset from
local technical terminology and traction power supply documents, evaluate
locally cached models on public and domain-oriented tasks, and test a QLoRA
adaptation path for selected candidates. The current local model cache contains
more models than the completed result export; the formal table in this draft
therefore distinguishes the planned model pool from the subset whose public
benchmark and efficiency results have already been exported.

The study is guided by four research questions:
\begin{enumerate}[leftmargin=1.5em]
  \item Which completed open LLM baselines provide the strongest public
  benchmark performance under a single RTX 3090 24GB constraint?
  \item How do bf16, int8, and int4 inference settings change memory footprint,
  latency, and generated-token throughput on the same machine?
  \item What does the current QLoRA adapter export show, and what must be rerun
  before making final domain-adaptation claims?
  \item Which models are most suitable for resource-constrained railway-domain
  deployment when accuracy, memory, and latency are considered together?
\end{enumerate}

The contributions are:
\begin{enumerate}[leftmargin=1.5em]
  \item We construct a railway-domain bilingual QA dataset with 22,164 training
  samples, 2,770 validation samples, and 2,770 test samples from local railway
  terminology and regulation material.
  \item We implement a reproducible single-GPU evaluation workflow that records
  public benchmark scores, local domain outputs, latency, throughput, and peak
  VRAM for multiple precision settings.
  \item We report the completed baseline and efficiency results currently
  available in the project CSV exports and clearly separate mature results from
  one-sample domain sanity checks.
  \item We curate a 90-entry BibTeX library and JSON literature-method matrix
  covering model families, resource-efficient LLM deployment, evaluation,
  PEFT, pruning, and federated adaptation.
\end{enumerate}

The rest of this paper follows the PeerJ Computer Science application-paper
structure. \Cref{sec:related_work} reviews model evaluation, efficient
adaptation, and resource-constrained deployment. \Cref{sec:problem_setting}
defines the railway application scenario. \Cref{sec:methodology} describes the
evaluation and adaptation workflow. \Cref{sec:experimental_setup} presents the
dataset, model pool, hardware, and metrics. \Cref{sec:results} reports the
current exported results, and \Cref{sec:discussion} discusses deployment
implications and limitations.
