\label{sec:methodology}

The methodology is designed to support deployment-oriented model selection
under a fixed hardware budget. It contains six stages: model-pool definition,
baseline evaluation, domain-task evaluation, efficiency measurement, QLoRA
adaptation, and multi-objective analysis.

\subsection{Model-Pool Definition}

The formal pool follows the latest project configuration rather than the older
partial result export. It contains ten local open models from Qwen, DeepSeek,
Mistral, Phi, Yi, Gemma, and GLM families. The pool intentionally mixes 4B--9B
models because a single RTX 3090 deployment must consider both capability and
operational cost. Models that are not yet evaluated are still listed in the
tables as placeholders, which keeps the final experiment matrix visible.

\subsection{Baseline and Public Evaluation}

Each model is loaded locally and evaluated without task-specific fine-tuning.
The public benchmark suite contains MMLU, C-Eval, GSM8K, and HumanEval. These
tasks measure broad knowledge, Chinese-oriented understanding, mathematical
reasoning, and code generation. The current draft reports exported values when
they exist and leaves missing ten-model int4 rows blank.

\subsection{Domain Evaluation}

Domain-QA and Domain-RegQA are evaluated with exact match, character-level F1,
token-level F1, reference containment, and length ratio. Exact match is useful
for deterministic terminology, but it is too strict for all valid translations.
Character-level F1 and token-level F1 provide softer matching, while reference
containment and length ratio reveal whether a model returns concise domain
answers or long off-target responses.

\subsection{Low-Bit Inference and Efficiency Measurement}

Baseline inference is evaluated under bf16, int8, and int4 where supported by
the local stack. The manuscript does not assume that quantization always
improves speed. It records peak allocated VRAM, peak reserved VRAM, latency,
and generated-token throughput, because backend kernels and dequantization
overhead can make low-bit inference slower even when memory usage improves.

\subsection{QLoRA Adaptation}

QLoRA is used as the adaptation mechanism because it freezes the base model,
loads the model in 4-bit form, and trains a small set of LoRA parameters. The
completed Adapter-A setting trains on Domain-QA. Planned variants include
Adapter-B on Domain-RegQA, Adapter-C on mixed Domain-QA + Domain-RegQA, and
Adapter-D on mixed data with completion-only loss masking. These variants
directly answer whether domain terminology, regulatory evidence, or mixed data
is more useful for this resource-constrained setting.

\subsection{Decision Rule}

The final recommendation is not the model with the largest public benchmark
score. A candidate is considered suitable only if it fits within 24GB VRAM,
has acceptable latency, preserves useful public benchmark quality, and improves
or maintains railway-domain metrics after adaptation. This decision rule
matches the intended local deployment scenario more closely than a single
leaderboard ranking.
